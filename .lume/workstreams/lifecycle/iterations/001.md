# Iteration 001 - Active-workstream selection: status field + current pointer

## DoD

- [x] 1. Current pointer: `.lume/current` is a one-line, human-readable file naming the current workstream slug. The engine resolves the workstream from it, not by alphabetical order. A missing or dangling pointer produces a clear error (exit 1) telling the operator to create/select one - never a silent fallback. (Evidence: with current -> lifecycle, `lume status` shows lifecycle though build-lume sorts first; remove/garble the pointer -> named error, exit 1.)
- [x] 2. Status field: `status: active|closed` in objective.md frontmatter is the single source of a workstream's lifecycle state, parsed deterministically. `objective_line()` and `lume status` keep working with frontmatter present - the `status:` line is never mistaken for the objective text. (Evidence: source + test; `lume status` prints the real objective first line.)
- [x] 3. `lume new <slug> "<title>"`: creates `.lume/workstreams/<slug>/objective.md` with `status: active` and the title, and sets `.lume/current` to <slug>. Refuses (non-zero, clear message) on an empty/invalid slug or one that already exists. (Evidence: scripted run creates a sandbox ws and flips current; re-running the same slug -> refusal.)
- [x] 4. `lume close`: sets the current workstream's status to closed and clears `.lume/current`. With no current, the engine refuses open/transition/status-load with a clear "no current workstream" error until a new current is set (via `lume new`). (Evidence: `lume close` then `lume open` -> named error, exit 1.)
- [x] 5. Migration: the two existing workstreams are brought into the new representation - build-lume objective.md gains `status: closed`, lifecycle gains `status: active`, and `.lume/current` is created pointing to lifecycle. `lume status` from the repo now resolves lifecycle. (Evidence: the three file states + `lume status` output.)
- [x] 6. Non-negotiables intact: no inference in the path; no commits; `new`/`close` touch only their own objective.md + `.lume/current` (plus the new dir for `new`); existing gate/transition/snapshot behaviour unchanged. (Evidence: git status after each verb; grep shows no model/network imports added.)
- [x] 7. Tests (stdlib unittest): cover pointer-based selection, missing/dangling-pointer error, status parsing (active + closed), closed-workstream refusal of open/transition, `lume new` (creates + sets current + refuses duplicate), `lume close` (flips field + de-currents). Existing 41 tests stay green; suite reports 0 failures. (Evidence: test count + run.)
- [x] 8. Orientation: `lume status` names the current workstream and lists the other workstreams with their status, so >1 workstream is visible at a glance and "which am I on" is answered directly. (Evidence: status output showing lifecycle as current + build-lume as closed.)
