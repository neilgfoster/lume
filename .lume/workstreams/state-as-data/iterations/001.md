## Self-review
Adversarial pass against the DoD:
- Inventory is grounded in the actual modules (workstream/iteration/plan/
  snapshot/frontmatter/repository read this session), not assumed. Caught and
  recorded that verdict outcomes are state currently encoded as prose, and that
  snapshot.md is already a derived view - both load-bearing for the design.
- Entity model is a *candidate*, deliberately leaving granularity (one file vs
  per-entity) open rather than over-committing in discovery.
- Honoured the constitution: flagged the dependency cost of validation against
  the engine's dependency-free ethos rather than assuming a library, and tied
  the JSON state back to the existing Repository contract (fork f) so this
  doesn't bypass the documented swap seam.
- Risk not yet resolved (correctly deferred): whether the migration buys back
  more time than it costs is a planning judgement, not a discovery output. Forks
  are explicit so planning has real decisions, not a foregone conclusion.

## Handback
Produced discovery.md (7 sections). Key finding: the authoritative machine state
is small and already isolated (status, the iteration frontmatter quad, verdict
outcomes, plan-item lines); everything else is prose artifacts or already-derived
views - so "state is data" is mostly extraction, not invention. snapshot.md
already proves the view-from-state pattern. Drafted a 3-entity JSON model,
chose JSON-Schema-at-the-boundary validation, sketched `entities`/`schema`/`get`
discovery verbs, and drew the state/artifact/view boundary. Six forks left for
planning (granularity, validation dependency, migration, DoD-as-data,
view regen, contract fit). No engine code touched.
