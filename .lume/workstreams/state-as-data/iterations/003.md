# Iteration 003 - P1: JSON Schemas + vendored validator

## DoD
- [x] Three JSON Schemas authored as engine data (draft-2020-12 subset) for
      `workstream`, `iteration`, `plan_item`, matching decisions.md's §2 shapes
      (workstream: slug/title/status/objective_artifact; iteration: id/type/phase/
      opened/title/verdicts[]/dod_artifact; plan_item: id/type/iter/tag/sketch).
      Enums (status, phase, type, tag, verdict) are constrained in-schema.
      (lume/schemas/{workstream,iteration,plan_item}.json)
- [x] A vendored, stdlib-only validator (no new dependency) that validates a dict
      against a schema and raises a single named boundary error (SchemaError in
      errors.py) with a message naming the entity + the offending field. Covers:
      required keys, type checks (string/integer/number/array/object/null/boolean),
      enum membership, and nested array-of-object items. (lume/validate.py)
- [x] Schemas are loadable by entity name (the seam a future `lume schema <entity>`
      verb will print): validate.load_schema(entity) + validate.entity_kinds().
- [x] Unit tests (stdlib unittest): each schema accepts a valid instance and
      rejects targeted invalid instances per failure mode (missing required, wrong
      type, bad enum, malformed/indexed verdict item, bool-is-not-int); validator
      error message asserted to name the field. (tests/test_validate.py, 11 tests)
- [x] Scope held: NO wiring into Repository/models/verbs and NO state.json read/
      write this iteration (those are P2-P4). Pure schema + validator + tests.
- [x] Full suite green: `cd .lume/engine && python3 -m unittest discover -s tests -t .`
      -> 88 tests OK.
- [x] No dead code / unused imports introduced.

## Self-review
Adversarial pass against the DoD:
- Schemas match decisions.md §2 exactly, including the two nullable unions
  (plan_item.iter, verdict.reason as ["...","null"]) - tested that null is
  accepted, not just tolerated.
- Hit the Python bool-is-int trap deliberately: `id: True` must fail an integer
  field. The validator excludes bool from integer/number and there's a test
  pinning it - this is the kind of silent-pass bug a naive isinstance check ships.
- Error messages name entity + dotted/indexed path (verdicts[0].verdict), proven
  by assertion, so a future boundary failure is debuggable, not opaque.
- Scope genuinely held: validate.py is imported by nothing yet; no model/verb/
  Repository change; no state.json written. The validator only loads schema files
  and checks dicts. P2 will wire it in.
- Validator is intentionally a subset (type/enum/required/properties/items) - it
  does NOT implement $ref, allOf, pattern, etc. That is the decision-(b) bet; if
  a schema later needs them, the swap point is documented in validate.py's
  docstring. Flagged so this isn't mistaken for a full JSON-Schema validator.
- Dependency-free confirmed: only stdlib (json, pathlib) imported.
