# state-as-data - decisions

Append-only. Newest at the bottom. Plan changes are logged here with their why.
Resolves the six forks in discovery.md §6.

- 2026-06-09 | (002 planning) GO / no-go = GO | Discovery showed the machine state is small and already isolated (status, the iteration quad, verdict outcomes, plan-item lines) and snapshot.md already proves view-from-state. Extraction cost is bounded; the payoff - deterministic agent interaction, schema-validated state, cold-start discovery - directly serves the orientation-first vision and the deterministic-over-inference non-negotiable. Added ceremony (3 schemas + a small validator + 3 read verbs) is one-time and small. Net positive against "ceremony must buy back its cost".
- 2026-06-09 | (002 planning) (a) Granularity = one state.json per workstream | The workstream is the unit of work, diff, and review; one validated object per workstream keeps a transition atomic (it writes the iteration field + recomputes derived in one file) and diffs cleanly at the boundary the operator reviews. Per-iteration JSON fragments an atomic transition; a repo-wide store loses per-workstream diffability. Prose stays in per-iteration markdown artifacts that state points at.
- 2026-06-09 | (002 planning) (b) Validation = vendored stdlib validator, schemas authored as JSON Schema | The engine is deliberately dependency-free (frontmatter.py reimplements YAML); the shapes are flat enums/strings/ints/arrays, a ~40-line validator covers them. Adding `jsonschema` for three flat schemas fails the minimal/buy-back test. Schemas are written as a JSON-Schema (draft 2020-12) subset so a real validator can drop in later if shapes grow (revisit (b) then).
- 2026-06-09 | (002 planning) (c) Migration = one-shot converter over ALL workstreams | A converter that reads existing markdown (frontmatter + verdict lines + plan-item lines) and emits state.json. Run on all three (2 closed + this one): the converter cost is paid once, the closed workstreams are free round-trip validation, and the queue keeps working uniformly. The dogfooding done-when requires this workstream migrated; doing all is marginal extra.
- 2026-06-09 | (002 planning) (d) DoD checkboxes = artifact, not data (v1) | DoD items are inference-authored prose; no gate branches on their checked state today. Parsing them into schema'd data adds cost for marginal value - fails buy-back. Revisit only if a gate ever needs machine-checkable DoD completion.
- 2026-06-09 | (002 planning) (e) View regen = automatic on every mutation (same path a render verb would call) | Zero-extra-step continuity is core: the snapshot must stay current at every boundary (this is what record_snapshot does now). On-demand-only regen would let views drift - the exact failure the snapshot exists to prevent. Auto-regen is cheap. A standalone `lume render` verb is optional/deferred, not needed for the value.
- 2026-06-09 | (002 planning) (f) Contract fit = JSON state lives behind the existing Repository seam | Repository becomes the loader/validator/persister of state.json and the view regenerator; models operate on validated data, not file parsing. Honours the design-for-replaceability non-negotiable - a future GitHub/Jira backing swaps Repository without touching models or verbs.
- 2026-06-09 | (006 P4) Plan change: P4 split into P4a (data-first model bridge, behaviour-preserving) + P4b (flip source of truth to state.json) | On starting P4, the monolith touched Iteration/Workstream/Repository/cli/snapshot + ~8 test files in one diff - too large to review safely in one short iteration, against the constitution's reviewable-iterations principle. The single-source-of-truth constraint makes the FLIP atomic, but the model prep is separable and behaviour-preserving. P4a adds Iteration.to_entity/from_entity + a canonical verdict parser (DRYs migrate) with no behaviour change; P4b consumes it to make state.json the truth with markdown as a regenerated view. Operator approved the split before any code was written. (Logged per decision e.) Note: the plan parser only matches numeric P<n> ids, so in plan.md the split is P4 (model bridge) + P5 (flip); the old P5/P6 became P6/P7. Caught because "P4a"/"P4b" were silently dropped from the parsed plan.
- 2026-06-09 | (005 P3) Migration normalises legacy iteration types: build -> execution, review -> closeout | P1's schema enforces the {discovery,planning,execution,closeout} vocabulary, but pre-typed iterations carry build (10) and review (1). build was the original hardcoded type the lifecycle workstream retired in favour of execution (its 005 note); build-lume's review/006 is literally "Retro ... and close build-lume" = closeout. Mapping them on migration is faithful to those documented supersessions, keeps the vocabulary clean (rather than polluting the enum with dead values), and the markdown artifact still carries the original frontmatter. Surfaced for operator review at 005 handback.
